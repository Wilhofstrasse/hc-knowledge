Spike complete. You ran all 5 user actions in PowerPoint today. Verdict: YELLOW — the simple watch-for-clicks approach works for three actions, doesn't apply to one, and has a real blind spot on one. The next phase needs a stronger safety net.

What passed

✓ 💬 Comment on §What passed
  • Typing into a shape. Every keystroke fired an event we could see. The tool catches it cleanly.
  • Pasting from another shape. One clean event when you pressed Cmd+V. Caught it.
  • Format painter (changing bold without changing text). No false alarm — the tool correctly ignored it because the words didn't change.

What was moot

✓ 💬 Comment on §What was moot
  • Undo across save and reopen. Cmd+Z did nothing after the deck was saved, closed, and reopened — that's how PowerPoint works on Mac (it forgets undo history on save). Bonus good news: the markers we put on shapes survived the save and reopen perfectly.

The blind spot (the YELLOW)

✓ 💬 Comment on §The blind spot (the YELLOW)

When another tool writes to a shape, we don't see it happen. You ran a small Script Lab snippet that programmatically replaced a shape's text. Zero events fired. The tool only noticed when you re-clicked the shape four minutes later — and even then, only because the text had changed compared to what we last saw. The tool's marker still said "AI wrote this", which is wrong: the user-controlled write should have flipped it. If we relied on this approach alone, the next AI fill on that shape would overwrite user content.

Locked recommendation for the next phase

✓ 💬 Comment on §Locked recommendation for the next phase

Make the tool look before it writes. Right before the AI is about to type into a shape, the tool first reads what's already there and compares it to what we last recorded. If they don't match, something changed without us noticing — leave the shape alone, treat it as your edit. This closes the blind spot for good. The watch-for-clicks approach stays useful as a friendly real-time signal ("you edited this, it's protected"), but it's no longer the thing keeping you safe.

Separate production bug found (caught and logged)

✓ 💬 Comment on §Separate production bug found (caught and logged)

During the spike I noticed our tool can't read the markers it just put on shapes — PowerPoint stores marker names in capital letters internally, and our code looks them up in lowercase. We have the side-by-side proof in today's logs. Small, separate fix; doesn't block the decision above.

Decisions waiting on you

✓ 💬 Comment on §Decisions waiting on you
  1. Should the AI always look before it writes? Yes means the tool always checks "has this shape changed since I last saw it?" before any AI fill — and leaves the shape alone if anything moved. This is the safety net that closes today's blind spot. No means we keep the simpler watch-for-clicks approach and accept the gap. My pick: yes.
  2. Should I fix the small marker-name bug we found today? Quick separate fix so the tool can read its own markers correctly. Yes (do it) or no (leave for later). My pick: yes.
  3. Should we also test the ten other PowerPoint actions the reviewer flagged? Things like Find & Replace, drag-and-drop, duplicating a slide and editing the duplicate, editing inside a group, the outline view, master/layout edits, content inside charts or tables, co-authoring sync from PowerPoint web, macOS dictation, and "reset slide". My pick: skip. If you say yes to decision 1, the safety check covers them all anyway — running the tests would only confirm what we already know. Yes (run them) or no (skip).

What this is. A multi-tenant PowerPoint add-in for consulting-grade decks. Consultants click a shape, the AI fills it with text grounded in real sources, every claim has a traffic-light, every action is logged for audit. Built for regulated industries — banking, insurance, asset management, pharma — where "show me where this number came from" is not optional.

Why this page exists. The full architecture has 10 invariants, 6 primitives, 6 user flows, and ~200 pages of locked specs across multiple files. That's right for engineers and Codex. It's wrong for the partner who needs the picture in 5 minutes. This page summarizes the architecture in three modules + one foundation, with links down for verification.

🖼️ Imagery LOCKED v2.1

Comment on §🖼️ Imagery LOCKED v2.1

All non-text visual content the AI fetches or proposes — cover photos, competitor/peer-firm logos for peer analysis, country flags + sector icons for PESTEL slides, peer-map firm icons. One module, broad scope.

6-tier picker for cover photos: user-upload → IR press kit → H&C curated library → licensed stock (Shutterstock + Getty) → AI-gen (illustrative only, NEVER fake real-firm building photos) → H&C placeholder.

Out of scope: employee/people photos (privacy + GDPR — manual upload only, not AI-fetched).

🔍 Grounding LOCKED v2.1 v2.1.1 pending

Comment on §🔍 Grounding LOCKED v2.1 v2.1.1 pending

The proof layer for every textual statement in the deck. Sources get ingested at deck open, retrieved via search, fed to the AI as grounded prompts, validated by structured-output schemas + semantic entailment, surfaced as traffic-lights + paragraph-level citations.

Filipe's model-agnostic feedback (#3) reshapes this: the spine is the universal citation envelope + entailment validator, NOT any one provider's API. Anthropic / Google / OpenAI become adapter optimizations.

Open: 15 Q-GR defaults, semantic entailment NLI eval set, multi-vendor cost attribution.
